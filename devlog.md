# 자율세특 개발 기록: 교실 원격 관리 시스템

## 프로젝트 개요

학교 컴퓨터실에서 사용하는 NetSupport School류 원격 관리 소프트웨어를 직접 구현하며, 정당한 관리 도구를 RAT(원격 접근 트로이목마)와 구분 짓는 보안 설계 요소를 실증적으로 탐구한다.

**핵심 연구 질문**: 동일한 원격 제어 아키텍처를 정당한 관리 도구로 유지시키는 최소 설계 요소는 무엇인가—그리고 그것을 코드로 어떻게 강제하는가?

**아키텍처**: 노트북(교사 서버) ↔ 데스크탑(학생 클라이언트), TCP 소켓 기반, 영상 채널(9999)과 제어 채널(9998)을 분리한 이중 채널 구조.

---

## M1 — 최소 화면 전송 파이프라인

### 목표
클라이언트(학생 PC) 화면을 캡처해 서버(교사 PC)로 실시간 전송, 화면에 표시.

### 구현 내용
- `mss` 라이브러리로 화면 캡처
- OpenCV(`cv2.imencode`)로 JPEG 압축
- TCP 소켓으로 전송 (4바이트 길이 헤더 + 데이터 순서, TCP 스트림 특성상 메시지 경계를 직접 관리해야 함)
- 수신 측에서 디코딩 후 `cv2.imshow`로 표시

### 문제 발견 및 해결

**1. 네트워크 구조 불일치**
- 증상: 두 기기가 ping조차 응답하지 않음
- 원인: 데스크탑이 공유기가 아닌 별도 장비(모뎀 직결로 추정)에 물려 있어 공인 IP(`59.5.163.176`)를 받고 있었음. 노트북은 사설 IP(`172.30.1.53`) 대역
- 해결: 데스크탑 랜선을 실제 공유기(GiGA WiFi home AX)의 LAN 포트로 재연결 → 같은 사설 대역(`172.30.1.x`)으로 통일
- **일반화**: 두 기기가 "네트워크에 연결되어 있다"는 사실만으로는 서로 통신 가능하다는 보장이 없다. 같은 서브넷(같은 게이트웨이)에 있어야 함

**2. ICMP(ping) 차단**
- 증상: 같은 대역인데도 ping 응답 없음
- 원인: Windows 방화벽이 기본적으로 ICMP 에코 요청을 차단
- 해결: 인바운드 규칙에서 "파일 및 프린터 공유(에코 요청 - ICMPv4-In)" 활성화
- **참고**: 이는 실제 TCP 통신과는 무관한 문제였음 — ping이 안 된다고 해서 우리가 쓸 TCP 소켓 통신도 안 되는 것은 아님. 별도로 확인 필요

**3. Windows 방화벽 인바운드 규칙 (포트별)**
- TCP 9999(영상 채널) 수신 측인 노트북에 인바운드 규칙 필요
- `netsh advfirewall firewall add rule name="ScreenServer-9999" dir=in action=allow protocol=TCP localport=9999`

### 결과
데스크탑 화면이 노트북 창에 거의 실시간(체감 딜레이 없음)으로 표시됨.

## M2 — 마우스/키보드 원격 제어 및 화면 잠금

### 목표
교사가 학생 PC의 마우스·키보드를 원격 조작하고, 필요시 화면을 잠글 수 있도록 확장.

### 설계 변경: 단방향 → 양방향 통신
M1은 클라이언트→서버 단방향(화면)만 존재했음. 제어 명령은 반대 방향(서버→클라이언트)이 필요해 통신 구조를 재설계함.

**채널 분리 방식 채택** (영상 9999 / 제어 9998, 각각 독립 소켓)을 선택한 이유: 하나의 소켓으로 양방향 메시지를 다중화(multiplexing)하는 것보다 구현이 단순하고, 디버깅 시 어느 채널의 문제인지 즉시 구분 가능.

### 문제 발견 및 해결

**1. 제어 큐 적체(queue backlog)로 인한 지연 및 입력 무반응**
- 증상: 마우스가 지연되어 따라오고, 키보드 입력이 거의 반영되지 않는 것처럼 느껴짐
- 원인 분석: `cv2.EVENT_MOUSEMOVE`는 마우스가 1픽셀만 움직여도 발생하는 고빈도 이벤트. 이를 제한 없이 전송 큐(`queue.Queue`)에 적재하자, 클릭·키 입력 명령이 이미 쌓인 수백 개의 이동 명령 뒤로 밀려 순서대로 처리되면서 체감상 "반응 없음"으로 나타남
- 해결: 마우스 이동 이벤트를 50ms(초당 최대 20회) 간격으로 스로틀링(throttle)하여 큐 적체를 방지
- **일반화**: 고빈도 이벤트를 큐 기반 순차 처리 구조에 그대로 흘려보내면, 우선순위가 낮은 이벤트가 우선순위 높은 이벤트(클릭, 키 입력)의 지연을 유발할 수 있음. 이는 이전 IDS 프로젝트에서 학습한 "실시간 시스템에서의 처리량-지연 트레이드오프"와 같은 계열의 문제

**2. DPI 배율 불일치로 인한 좌표 오차**
- 증상: 두 기기의 디스플레이 배율이 다를 때(노트북 125%, 데스크탑 100%) 클릭 위치가 의도한 지점과 어긋남
- 원인: Windows는 DPI 인식(DPI-aware)을 선언하지 않은 프로세스에 대해 좌표를 자동으로 가상화(배율에 맞춰 재계산)한다. `mss`의 화면 캡처 좌표(물리 픽셀 기준)와 `pyautogui`가 해석하는 좌표계(비-DPI-aware 상태에서는 가상화된 좌표)가 불일치하면서 오차 발생
- 해결: 프로그램 시작 시 `ctypes.windll.shcore.SetProcessDpiAwareness(2)`(PROCESS_PER_MONITOR_DPI_AWARE)를 호출해 각 프로세스가 항상 실제 물리 픽셀 좌표를 사용하도록 강제
- **설계 판단**: 배율 차이를 수치로 하드코딩해 보정하는 방식(예: 100/125 배율 곱)도 가능했으나, 학교 컴퓨터실의 PC마다 배율 설정이 상이할 수 있어 범용성이 떨어진다고 판단. 근본 원인(DPI 가상화)을 API 수준에서 제거하는 방식을 택함

**3. 화면 잠금의 실효성 문제**
- 최초 구현: 전체화면 검은 오버레이 창(tkinter)만으로 "잠금"을 시각적으로만 구현 → 실제 입력은 차단되지 않아 학생이 Alt+Tab 등으로 우회 가능한 한계 존재
- 개선: Windows API `user32.dll`의 `BlockInput()`을 호출해 실제 마우스·키보드 입력을 시스템 레벨에서 차단. 오버레이 창은 시각적 피드백 용도로 병행 유지
- **주목할 설계 제약**: `BlockInput()`은 Ctrl+Alt+Del(보안 주의 시퀀스) 입력만큼은 차단하지 못한다. 이는 Windows가 의도적으로 설계한 예외로, 어떤 애플리케이션도 사용자의 최후 탈출 수단(작업 관리자 강제 종료 등)을 완전히 봉쇄할 수 없도록 하는 OS 차원의 안전장치다
- **안전장치 추가**: `BlockInput(True)` 호출 후 프로그램이 비정상 종료되면 입력이 영구히 차단된 상태로 남을 위험이 있음. 이를 방지하기 위해 `try...finally` 구문으로 프로그램 종료 시(정상/비정상 무관) 반드시 `BlockInput(False)`가 호출되도록 안전장치를 구현
- `BlockInput()`은 관리자 권한 프로세스에서만 정상 동작하므로, 실행 시 관리자 권한 여부를 확인하고 아닐 경우 경고를 출력하도록 처리

### 결과
마우스 이동/클릭, 키보드 입력, 화면 잠금(L 키 토글) 모두 정상 동작 확인. 약간의 마우스 이동 지연은 남아 있으며 추후 개선 과제로 보류(영상 채널 프레임률/화질 조정, 스로틀 간격 조정 등 검토 예정).

## 다음 단계 (M3 예정)

현재 구조는 서버 1대 : 클라이언트 1대로 제한됨. 실제 교실 환경(학생 PC 다수)을 지원하려면:
- 서버가 다중 클라이언트 연결을 동시에 수락(accept)하고 클라이언트별 스레드로 관리
- 화면을 그리드 형태로 동시 표시
- 제어 대상 클라이언트를 선택하는 인터페이스 필요

## 정당성 관련 설계 원칙 (연구 질문에 대한 중간 결론)

M1~M2를 거치며 이 시스템이 "정당한 교실 관리 도구"로 남기 위해 반드시 필요하다고 판단한 요소:

1. **가시적 동의 표시** — 제어당하는 화면에 상시 인디케이터 표시 (구현 예정, M2에서는 잠금 시에만 표시됨)
2. **인증** — 아무 클라이언트나 서버에 접속하지 못하도록 사전 공유 키 검증 (구현 예정)
3. **감사 로그** — 언제, 누가, 어떤 학생 PC에 어떤 조작(조회/제어/잠금)을 했는지 기록 (구현 예정)

동일한 소켓·화면캡처·입력제어 코드라도, 이 세 요소의 유무가 "교실 관리 도구"와 "은닉형 감시 도구"를 가르는 지점이라는 것이 현재까지의 실증적 결론이다. 특히 `BlockInput`처럼 강력한 API일수록, 오작동 시 사용자에게 미치는 피해가 크기 때문에 안전장치(예외 처리, 강제 해제 로직)를 코드 레벨에서 명시적으로 설계해야 한다는 점을 M2에서 직접 확인했다.
