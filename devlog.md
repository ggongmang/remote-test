# 자율세특 개발 기록: 교실 원격 관리 시스템

## 프로젝트 개요

학교 컴퓨터실에서 사용하는 NetSupport School류 원격 관리 소프트웨어를 직접 구현하며, 정당한 관리 도구를 RAT(원격 접근 트로이목마)와 구분 짓는 보안 설계 요소를 실증적으로 탐구한다.

**핵심 연구 질문**: 동일한 원격 제어 아키텍처를 정당한 관리 도구로 유지시키는 최소 설계 요소는 무엇인가—그리고 그것을 코드로 어떻게 강제하는가?

**아키텍처**: 노트북(교사 서버) ↔ 데스크탑(학생 클라이언트), TCP 소켓 기반, 영상 채널(9999)과 제어 채널(9998)을 분리한 이중 채널 구조.

---

## M1 — 최소 화면 전송 파이프라인

### 목표
클라이언트(학생 PC) 화면을 캡처해 서버(교사 PC)로 실시간 전송, 화면에 표시.

### 구현 내용
- `mss` 라이브러리로 화면 캡처
- OpenCV(`cv2.imencode`)로 JPEG 압축
- TCP 소켓으로 전송 (4바이트 길이 헤더 + 데이터 순서, TCP 스트림 특성상 메시지 경계를 직접 관리해야 함)
- 수신 측에서 디코딩 후 `cv2.imshow`로 표시

### 문제 발견 및 해결

**1. 네트워크 구조 불일치**
- 증상: 두 기기가 ping조차 응답하지 않음
- 원인: 데스크탑이 공유기가 아닌 별도 장비(모뎀 직결로 추정)에 물려 있어 공인 IP(`59.5.163.176`)를 받고 있었음. 노트북은 사설 IP(`172.30.1.53`) 대역
- 해결: 데스크탑 랜선을 실제 공유기(GiGA WiFi home AX)의 LAN 포트로 재연결 → 같은 사설 대역(`172.30.1.x`)으로 통일
- **일반화**: 두 기기가 "네트워크에 연결되어 있다"는 사실만으로는 서로 통신 가능하다는 보장이 없다. 같은 서브넷(같은 게이트웨이)에 있어야 함

**2. ICMP(ping) 차단**
- 증상: 같은 대역인데도 ping 응답 없음
- 원인: Windows 방화벽이 기본적으로 ICMP 에코 요청을 차단
- 해결: 인바운드 규칙에서 "파일 및 프린터 공유(에코 요청 - ICMPv4-In)" 활성화
- **참고**: 이는 실제 TCP 통신과는 무관한 문제였음 — ping이 안 된다고 해서 우리가 쓸 TCP 소켓 통신도 안 되는 것은 아님. 별도로 확인 필요

**3. Windows 방화벽 인바운드 규칙 (포트별)**
- TCP 9999(영상 채널) 수신 측인 노트북에 인바운드 규칙 필요
- `netsh advfirewall firewall add rule name="ScreenServer-9999" dir=in action=allow protocol=TCP localport=9999`

### 결과
데스크탑 화면이 노트북 창에 거의 실시간(체감 딜레이 없음)으로 표시됨.

## M2 — 마우스/키보드 원격 제어 및 화면 잠금

### 목표
교사가 학생 PC의 마우스·키보드를 원격 조작하고, 필요시 화면을 잠글 수 있도록 확장.

### 설계 변경: 단방향 → 양방향 통신
M1은 클라이언트→서버 단방향(화면)만 존재했음. 제어 명령은 반대 방향(서버→클라이언트)이 필요해 통신 구조를 재설계함.

**채널 분리 방식 채택** (영상 9999 / 제어 9998, 각각 독립 소켓)을 선택한 이유: 하나의 소켓으로 양방향 메시지를 다중화(multiplexing)하는 것보다 구현이 단순하고, 디버깅 시 어느 채널의 문제인지 즉시 구분 가능.

### 문제 발견 및 해결

**1. 제어 큐 적체(queue backlog)로 인한 지연 및 입력 무반응**
- 증상: 마우스가 지연되어 따라오고, 키보드 입력이 거의 반영되지 않는 것처럼 느껴짐
- 원인 분석: `cv2.EVENT_MOUSEMOVE`는 마우스가 1픽셀만 움직여도 발생하는 고빈도 이벤트. 이를 제한 없이 전송 큐(`queue.Queue`)에 적재하자, 클릭·키 입력 명령이 이미 쌓인 수백 개의 이동 명령 뒤로 밀려 순서대로 처리되면서 체감상 "반응 없음"으로 나타남
- 해결: 마우스 이동 이벤트를 50ms(초당 최대 20회) 간격으로 스로틀링(throttle)하여 큐 적체를 방지
- **일반화**: 고빈도 이벤트를 큐 기반 순차 처리 구조에 그대로 흘려보내면, 우선순위가 낮은 이벤트가 우선순위 높은 이벤트(클릭, 키 입력)의 지연을 유발할 수 있음. 이는 이전 IDS 프로젝트에서 학습한 "실시간 시스템에서의 처리량-지연 트레이드오프"와 같은 계열의 문제

**2. DPI 배율 불일치로 인한 좌표 오차**
- 증상: 두 기기의 디스플레이 배율이 다를 때(노트북 125%, 데스크탑 100%) 클릭 위치가 의도한 지점과 어긋남
- 원인: Windows는 DPI 인식(DPI-aware)을 선언하지 않은 프로세스에 대해 좌표를 자동으로 가상화(배율에 맞춰 재계산)한다. `mss`의 화면 캡처 좌표(물리 픽셀 기준)와 `pyautogui`가 해석하는 좌표계(비-DPI-aware 상태에서는 가상화된 좌표)가 불일치하면서 오차 발생
- 해결: 프로그램 시작 시 `ctypes.windll.shcore.SetProcessDpiAwareness(2)`(PROCESS_PER_MONITOR_DPI_AWARE)를 호출해 각 프로세스가 항상 실제 물리 픽셀 좌표를 사용하도록 강제
- **설계 판단**: 배율 차이를 수치로 하드코딩해 보정하는 방식(예: 100/125 배율 곱)도 가능했으나, 학교 컴퓨터실의 PC마다 배율 설정이 상이할 수 있어 범용성이 떨어진다고 판단. 근본 원인(DPI 가상화)을 API 수준에서 제거하는 방식을 택함

**3. 화면 잠금의 실효성 문제**
- 최초 구현: 전체화면 검은 오버레이 창(tkinter)만으로 "잠금"을 시각적으로만 구현 → 실제 입력은 차단되지 않아 학생이 Alt+Tab 등으로 우회 가능한 한계 존재
- 개선: Windows API `user32.dll`의 `BlockInput()`을 호출해 실제 마우스·키보드 입력을 시스템 레벨에서 차단. 오버레이 창은 시각적 피드백 용도로 병행 유지
- **주목할 설계 제약**: `BlockInput()`은 Ctrl+Alt+Del(보안 주의 시퀀스) 입력만큼은 차단하지 못한다. 이는 Windows가 의도적으로 설계한 예외로, 어떤 애플리케이션도 사용자의 최후 탈출 수단(작업 관리자 강제 종료 등)을 완전히 봉쇄할 수 없도록 하는 OS 차원의 안전장치다
- **안전장치 추가**: `BlockInput(True)` 호출 후 프로그램이 비정상 종료되면 입력이 영구히 차단된 상태로 남을 위험이 있음. 이를 방지하기 위해 `try...finally` 구문으로 프로그램 종료 시(정상/비정상 무관) 반드시 `BlockInput(False)`가 호출되도록 안전장치를 구현
- `BlockInput()`은 관리자 권한 프로세스에서만 정상 동작하므로, 실행 시 관리자 권한 여부를 확인하고 아닐 경우 경고를 출력하도록 처리

### 결과
마우스 이동/클릭, 키보드 입력, 화면 잠금(L 키 토글) 모두 정상 동작 확인. 약간의 마우스 이동 지연은 남아 있으며 추후 개선 과제로 보류(영상 채널 프레임률/화질 조정, 스로틀 간격 조정 등 검토 예정).

## M3 — 다중 클라이언트 동시 모니터링 및 선택 제어

### 목표
서버 1대 : 클라이언트 1대로 제한되어 있던 구조를, 실제 교실처럼 서버 1대가 다수의 학생 PC를 동시에 관리할 수 있도록 확장.

### 설계
- **영상 채널**: `accept()`를 반복 호출하는 별도 스레드(`video_accept_loop`)를 두어, 새 클라이언트가 연결될 때마다 고유 ID(`cid`)를 부여하고 전용 수신 스레드를 생성. 클라이언트별 최신 프레임은 `clients` 딕셔너리에 스레드 안전하게 저장
- **제어 채널**: 클라이언트마다 별도의 제어 연결 및 전송 큐를 가짐 (M2의 단일 큐 구조를 클라이언트 수만큼 확장)
- **화면 구성**: 두 개의 창으로 분리
  - `Classroom Monitor`: 전체 학생 화면을 축소한 썸네일 그리드로 표시 (읽기 전용, 선택된 학생은 초록 테두리로 강조)
  - `Control`: 숫자 키(1~9)로 선택한 학생의 화면만 원본 크기로 표시, M2와 동일한 마우스/키보드/잠금 제어 적용

### client.py는 변경 없음
클라이언트 입장에서는 서버에 연결하는 방식이 M2와 동일하므로, 여러 대의 `client.py`가 각자 독립적으로 실행되어도 서버의 accept 반복문이 알아서 순차적으로 받아들인다. 이는 서버-클라이언트 아키텍처에서 "연결 수용 로직을 서버 쪽에 집중시키면 클라이언트 코드는 그대로 재사용 가능하다"는 설계 원칙을 보여줌.

### 현재 버전의 한계 (문서화)
- 학생 선택을 숫자 키(1~9)로 구현해, 현재는 최대 9명까지만 개별 선택 가능. 실제 교실(30명 내외) 규모로 확장하려면 클릭 기반 선택 방식으로 개선이 필요함 — 다음 단계 과제로 남김
- 로컬 환경(기기 2대)의 한계로 실제 다중 물리 PC 동시 연결 테스트는 학교 컴퓨터실에서 진행 예정. 현재는 그리드 표시 및 선택 제어 로직 자체의 정상 동작만 노트북-데스크탑 1:1 연결로 확인함

### 결과
데스크탑 1대를 클라이언트로 연결한 상태에서 그리드 표시, 숫자 키 선택, Control 창 전환, 마우스/키보드/잠금 제어, ESC를 통한 선택 해제까지 모두 정상 동작 확인.

## M4 — 인증 / 상시 동의 표시 / 감사 로그

### 목표
연구 질문("동일한 원격 제어 아키텍처를 정당한 관리 도구로 유지시키는 최소 설계 요소는 무엇인가")에 대한 직접적인 답으로, M1~M3에서 미뤄뒀던 세 가지 보안·윤리적 요소를 구현.

### 1. 인증 (공유 비밀키)
- 클라이언트가 서버에 접속하면, 영상 데이터를 보내기 전에 먼저 사전 공유된 `AUTH_KEY` 문자열을 JSON으로 전송
- 서버는 이를 확인해 일치하지 않으면 즉시 연결을 끊고 `AUTH_FAIL`로 로그에 기록. 일치할 때만 학생으로 등록하고 정상적인 화면 수신을 시작
- **구현상 문제**: 인증 메시지(줄바꿈 종료)와 이후의 영상 프레임(4바이트 길이 헤더 기준)은 서로 다른 방식으로 메시지 경계를 구분함. TCP는 스트림이라 두 종류의 메시지가 소켓 버퍼에서 섞여 도착할 수 있어, 이를 처리하기 위해 내부 버퍼를 직접 관리하는 `ConnReader` 클래스를 작성함 (`read_line`으로 인증 메시지를, `read_exact`로 이후 프레임 데이터를 같은 버퍼에서 순서대로 읽음)
- **일반화**: 하나의 소켓 연결 안에서 프로토콜이 여러 단계(핸드셰이크 → 데이터 전송)로 나뉠 때는, 각 단계의 메시지 경계 구분 방식이 다르더라도 하나의 버퍼로 일관되게 관리해야 데이터 유실이나 오작동을 막을 수 있음

### 2. 상시 가시적 동의 표시
- M2까지는 "잠금" 상태일 때만 오버레이가 떴음. M4에서는 **서버에 정상 연결(인증 성공)되어 있는 동안 항상** 화면 상단에 작은 배너("선생님이 이 화면을 보고 있습니다")가 표시되도록 별도의 tkinter 창을 추가
- 잠금 오버레이(전체화면, 상태에 따라 토글)와 동의 배너(작은 크기, 연결 상태에 따라 토글)는 서로 독립적인 창으로 구현, 각각의 상태 변수를 폴링하는 방식 사용
- **알려진 한계**: 이 배너는 시각적 표시일 뿐이며, 사용자가 다른 창으로 가리거나 포커스를 옮기는 것 자체를 막지는 못함. 진짜로 "항상 최상단에서 가려지지 않게" 만들려면 더 강한 창 관리 API가 필요하며, 이는 후속 개선 과제로 남김

### 3. 감사 로그
- `audit_log.csv`에 연결(CONNECT), 해제(DISCONNECT), 인증실패(AUTH_FAIL), 학생 선택(SELECT), 잠금 토글(LOCK_TOGGLE) 이벤트를 타임스탬프와 함께 기록
- **로그 granularity에 대한 설계 판단**: 마우스 이동처럼 초당 수십 건씩 발생하는 이벤트까지 전부 기록하면 로그가 무의미하게 비대해지고 정작 중요한 이벤트(누가 언제 어떤 학생을 제어했는가)를 찾기 어려워짐. 따라서 "의미 있는 행위 단위"만 기록하도록 범위를 제한함 — 실제 감사 로그 시스템 설계에서도 동일한 트레이드오프가 존재함
- CSV를 택한 이유: 이전 IDS 프로젝트에서 SQLite를 사용하며 겪었던 동시 쓰기 오류(OneDrive 동기화 간섭) 경험을 고려해, 이번 단계에서는 단순하고 디버깅이 쉬운 방식을 우선함. 추후 여러 서버 인스턴스나 대규모 로그 조회가 필요해지면 SQLite로 전환하는 것을 개선 과제로 남김

### 결과
AUTH_KEY 불일치 시 연결이 정상적으로 거부되고 로그에 기록됨을 확인. 인증 성공 시 데스크탑 화면에 동의 배너가 상시 노출됨을 확인. `audit_log.csv`에 CONNECT/SELECT/LOCK_TOGGLE 이벤트가 시간순으로 정상 기록됨을 확인.

### 정당성 설계 원칙 검증 완료
M2 시작 시점에 "예정"으로 남겨뒀던 세 요소(가시적 동의 표시, 인증, 감사 로그)가 모두 구현되어, 앞서 세운 연구 질문에 대한 실증적 근거가 M1~M4 전 과정에 걸쳐 축적됨.

## M5 — 실측 (지연시간 / FPS / 대역폭)

### 측정 방법
클라이언트가 프레임 전송 시 8바이트 타임스탬프(`time.time()`)를 헤더에 함께 실어 보내고, 서버가 수신 시각과의 차이로 지연시간을 계산. 30프레임(약 3초)마다 평균 지연시간, FPS, 대역폭(KB/s), 동시접속 수를 `metrics_log.csv`에 기록.

### 측정 결과 (홈 환경, 노트북-데스크탑 2대)
- 동시접속 1명: 평균 "지연" ≈ 1090ms, FPS ≈ 10.0, 대역폭 ≈ 1325KB/s
- 동시접속 2명(같은 데스크탑에서 client.py 2개 실행): 평균 "지연" ≈ 1088ms, FPS ≈ 10.0, 대역폭 ≈ 1325KB/s (학생당)

### 중요한 발견: 측정값의 타당성 재검토
지연시간이 약 1090ms로 거의 고정된 값으로 측정되었으나, 이는 실제 네트워크 지연이라 보기 어려움. 근거:
1. 별도로 측정한 ICMP ping 왕복시간이 평균 6ms에 불과했고
2. M1 단계에서 체감상 지연이 거의 없다고 확인했으며
3. 무엇보다 측정값의 변동폭이 매우 작아(1086~1093ms), 실제 네트워크 지연에서 기대되는 변동성과 맞지 않음

**결론**: 이 측정 방식(클라이언트 전송 시각 vs 서버 수신 시각의 단순 차이)은 두 기기의 시스템 시계가 정확히 동기화되어 있다는 전제가 필요한데, 이번 실험에서는 그 전제가 충족되지 않아 노트북-데스크탑 간 시계 오차(약 1.09초)가 "지연시간"으로 잘못 측정된 것으로 판단됨. 이는 시간 동기화가 보장되지 않은 별도 기기 간 단방향 지연을 측정할 때 흔히 발생하는 방법론적 함정이며, 정확한 측정을 위해서는 NTP 수준의 시계 동기화 또는 왕복시간(RTT) 기반 측정으로 전환이 필요함 — 향후 개선 과제로 남김

**그럼에도 유효한 결론**: 시계 오차가 일정하다는 전제 하에, 동시접속 인원에 따른 상대적 비교는 유효함. 1명 → 2명으로 증가해도 평균 "지연" 측정값에 유의미한 변화가 없었으므로, 이번 규모(2명)에서는 동시접속 증가가 지연에 미치는 영향이 미미하다고 잠정 결론 지을 수 있음

### 대역폭 분석 (신뢰 가능한 데이터)
타임스탬프 문제와 무관하게, 전송 바이트 수는 정확히 측정됨. 학생 1인당 약 1.3MB/s의 대역폭이 소요됨을 확인. 이를 실제 교실 규모(약 30명)로 단순 외삽하면 약 39MB/s의 동시 트래픽이 발생할 수 있어, 학교 네트워크 인프라가 이를 감당 가능한지가 M6(실제 컴퓨터실 테스트) 이전에 고려해야 할 요소로 확인됨. 필요시 JPEG 품질(현재 60) 또는 목표 FPS(현재 10)를 낮춰 대역폭을 절감하는 방안을 검토할 수 있음

## 보안 취약점 분석 및 대응 (M6 이전 보강)

### 발견된 취약점
시스템을 공격자(학생) 관점에서 재검토하여 다음 취약점을 식별함:

1. **인증키 평문 하드코딩** — `client.py` 소스코드를 열람하면 `AUTH_KEY`를 그대로 읽을 수 있음
2. **통신 전 구간 비암호화** — Wireshark 등으로 네트워크를 스니핑하면 인증키, 화면, 키 입력이 그대로 노출됨
3. **서버 신원 미검증** — 클라이언트만 인증하고 서버 쪽 신원은 확인하지 않아, 학생이 가짜 서버를 띄워 다른 학생을 낚을 수 있는 여지가 있음
4. **BlockInput의 Ctrl+Alt+Del 우회** — OS가 의도적으로 허용하는 예외이므로 소프트웨어로 막을 수 없음 (설계 원칙으로 이미 인지하고 있던 부분)
5. **관리자 권한 부재 시 잠금 무력화** — BlockInput은 조용히 실패하고 경고만 출력하므로, 관리자 권한 없이 실행되면 "잠긴 줄 알았는데 실제로는 안 잠긴" 상태가 될 수 있음
6. **포트 스캔을 통한 정찰 가능성** — 인증에 실패해도 해당 포트에 서비스가 떠 있다는 사실 자체는 노출됨

### 대응 1: TLS 암호화 적용 (영상 채널)
- 자체 서명 인증서(`gen_cert.py`로 생성)를 서버에 적용, `ssl.SSLContext`로 영상 채널(9999) 전체를 TLS로 감쌈
- 클라이언트는 서버의 인증서를 미리 `server.crt` 파일로 전달받아 신뢰하는 **TOFU(Trust On First Use)** 방식 채택 — 공인 CA 인증이 아니라 "이 특정 인증서를 신뢰한다"는 방식으로, 사설망 프로토타입 환경에 적합하다고 판단
- **핵심 통찰**: 인증키 교환이 영상 채널에서 이루어지므로, 이 채널을 암호화하는 것만으로 취약점 1번과 2번이 동시에 해결됨 — 낮은 계층(전송 계층)의 보안을 강화하면 상위의 여러 문제가 한 번에 해결되는 사례
- **남은 한계**: 제어 채널(9998)은 통신 방향이 반대(서버가 클라이언트에 접속)라서 TLS 서버/클라이언트 역할이 꼬이는 문제가 있어 이번 단계에서는 암호화하지 못함. 이를 해결하려면 제어 채널도 클라이언트가 서버에 접속하는 방향으로 아키텍처를 재설계해야 함 — 후속 과제로 남김

### 대응 2: 타이밍 공격 방지
- 인증키 비교를 `==` 대신 `hmac.compare_digest()`로 교체하여, 비교에 걸리는 시간차를 이용해 키를 한 글자씩 유추하는 타이밍 공격을 원천 차단

### 실제 배포 전 안정성 보강
- **클라이언트 자동 재연결**: 기존에는 연결이 끊기면 프로그램이 완전히 종료되어 학생이 수동으로 재실행해야 했음. 네트워크 일시 단절 시 5초 간격으로 자동 재연결을 시도하도록 개선 (단, 인증 실패는 설정 오류이므로 재시도하지 않고 종료)
- **그리드 화면 크기 제한**: 학생 수가 많아질수록(실제 교실 약 30명) 그리드 이미지가 무한정 커져 교사 화면 밖으로 벗어날 수 있음. 최대 높이를 제한하고 초과 시 비율에 맞게 축소, 클릭 좌표도 축소 비율을 반영해 정확한 학생이 선택되도록 보정

### 학교 테스트 전 확인이 필요한 비-코드 요소
코드 완성도와 별개로, 다음은 학교 컴퓨터실 현장에서만 확인 가능하여 사전에 정보 선생님께 문의 필요:
1. 학생 PC 간 로컬 네트워크 통신 허용 여부 (VLAN 등으로 격리되어 있을 가능성)
2. 학생 계정의 관리자 권한 실행 가능 여부 (BlockInput, 방화벽 규칙에 필요)
3. Python 및 필요 라이브러리 설치 가능 여부

### 실전에서 드러난 배포/설정 동기화 문제
TLS 적용 후 홈 환경 재검증 중 다음 문제를 실제로 겪음:
- **동적 IP 변경**: 공유기 재연결로 노트북(서버) IP가 바뀌어 클라이언트 접속 실패. 원인 진단 후 클라이언트의 `SERVER_IP` 값을 갱신하여 해결. 근본적으로는 서버 기기에 고정 IP(공유기의 DHCP 예약)를 설정해두는 것이 재발 방지책
- **인증서 버전 불일치**: `gen_cert.py`를 여러 번 실행하면서 노트북의 `server.crt`가 갱신되었으나, 데스크탑에는 이전 버전의 인증서가 남아있어 TLS 인증서 검증 실패(`CERTIFICATE_VERIFY_FAILED`) 발생. `Get-FileHash`로 두 기기의 파일 해시를 비교해 불일치를 확인하고 최신 파일을 재전달하여 해결
- **파일 인코딩 문제**: PowerShell의 `Get-Content`/`Set-Content`로 소스 코드(한글 주석 포함)를 편집하자 기본 인코딩이 UTF-8이 아니어서 파일이 깨져 `SyntaxError` 발생. 이후로는 코드 내용을 직접 텍스트 치환하는 대신 파일을 통째로 재배포하는 방식으로 회피

**일반화 및 향후 과제**: 이번 사례들은 파일 하나(설정, 인증서)를 수동으로 여러 기기에 복사하는 배포 방식이 기기 수가 늘어날수록 버전 불일치 위험이 커진다는 것을 보여줌. 학교 컴퓨터실처럼 수십 대 규모로 확장하려면, 수동 파일 복사 대신 공유 네트워크 드라이브나 배포 스크립트를 활용하는 방안을 검토할 필요가 있음

## 정당성 관련 설계 원칙 (연구 질문에 대한 중간 결론)

M1~M2를 거치며 이 시스템이 "정당한 교실 관리 도구"로 남기 위해 반드시 필요하다고 판단한 요소:

1. **가시적 동의 표시** — 제어당하는 화면에 상시 인디케이터 표시 (구현 예정, M2에서는 잠금 시에만 표시됨)
2. **인증** — 아무 클라이언트나 서버에 접속하지 못하도록 사전 공유 키 검증 (구현 예정)
3. **감사 로그** — 언제, 누가, 어떤 학생 PC에 어떤 조작(조회/제어/잠금)을 했는지 기록 (구현 예정)

동일한 소켓·화면캡처·입력제어 코드라도, 이 세 요소의 유무가 "교실 관리 도구"와 "은닉형 감시 도구"를 가르는 지점이라는 것이 현재까지의 실증적 결론이다. 특히 `BlockInput`처럼 강력한 API일수록, 오작동 시 사용자에게 미치는 피해가 크기 때문에 안전장치(예외 처리, 강제 해제 로직)를 코드 레벨에서 명시적으로 설계해야 한다는 점을 M2에서 직접 확인했다.
